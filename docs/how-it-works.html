Zero context bloat.

mcp-exec proxies all downstream MCP servers through a sandboxed execution layer. Only the final result enters the context window — intermediate data never does.

The Problem

Every connected MCP server dumps its full JSON schema into Claude's context at startup. A 7-server setup — Slack, GitHub, Gmail, Calendar, Linear, Salesforce, Notion — costs 143,000 tokens before you've said a word. Tool call results add another 10–40k per round trip.

The Solution

tools() returns 100-token summaries instead of full schemas. exec() runs multi-tool MCP workflows inside an OS-level sandbox — the code runs, tools get called, data gets processed, and only the shaped result returns to Claude.

The Payoff

43,800 tokens becomes 90 tokens. Not a compression trick — intermediate data never enters context at all. Multi-step workflows that would overflow a 200k window now fit in a paragraph. Schema bloat disappears.

Startup sequence: mcp-exec reads .claude/mcp.json → connects all downstream servers via connectMcpClients() → fetches their tool lists with buildCatalog() → registers Node.js loader hooks with the tool catalog → starts the HTTP bridge on 127.0.0.1:{random_port} → creates a SessionManager → registers tools and exec with the MCP SDK. From this point, Claude only sees two tools. Everything else is internal.
Startup sequence — step by step
1

Read sandbox config

Merge ~/.claude/settings.json and .claude/settings.json sandbox blocks. Map to SandboxRuntimeConfig. Extract MCP server hostnames from .claude/mcp.json and add to allowedDomains. Initialize SandboxManager (srt).

2

Connect downstream MCP clients

Read .claude/mcp.json, filter out mcp-exec itself. For each remaining server: create Client + StdioClientTransport, call client.connect() via Promise.allSettled. Failed servers go to unavailable[] — they don't crash startup.

3

Build tool catalog

Call client.listTools() on each connected server via Promise.allSettled. For each tool: derive a human-readable signature from inputSchema.properties and the required array. Store as ToolSummary. Servers whose listTools() fails join unavailable[].

// deriveSignature output for github/listPullRequests: "listPullRequests(owner: string, repo: string, state?: string, per_page?: number): unknown"
4

Register Node.js loader hooks

Call module.register('./loader/hooks.js', { data: { toolsByServer, unavailableServers } }). The loader hooks worker initializes with the catalog. From this point, any import('mcp/server-name') in sandboxed Node code is intercepted and returns a generated shim module — no disk I/O, no real package resolution.

5

Start HTTP bridge

Create BridgeServer with all MCP clients. Call bridge.start() — binds to 127.0.0.1:0 (OS assigns port). The port is then injected into every generated Python MCP package as _BRIDGE = "http://127.0.0.1:{port}/call".

6

Register tools with MCP SDK and go live

Create SessionManager. Create exec dispatcher via createExecDispatcher(sessions, clients, bridge, toolsByServer). Register tools and exec tool handlers on the MCP Server. Connect via StdioServerTransport. Claude Code can now call the two tools.

Catalog search algorithm
// searchTools("create issue linear") // Step 1: tokenize, drop stop words tokens = ['create', 'issue', 'linear'] // Step 2: for each tool, build search text: // description.toLowerCase() + camelCase-split(name) toolText = "create a new issue in a linear team create issue" // Step 3: every token must appear in toolText return catalog.filter(tool => tokens.every(t => toolText(tool).includes(t)) );
camelCaseTokens: "listPullRequests" splits to ["list", "pull", "requests"]. So tools("pull") matches it even without the exact name. Stop words (a, an, the, and, or, in, to, of, for, with) are filtered before matching.
UnavailableServer handling
// tools("*") output when a server failed to connect: [ { "server": "github", "name": "listPullRequests", "description": "...", "signature": "..." }, // ... other working tools ... { "server": "notion", "status": "unavailable", "reason": "not configured in mcp.json" } ]
Partial availability: A single unavailable server never crashes mcp-exec. Promise.allSettled is used at every async boundary — connection, tool listing, and exec dispatch all handle partial failures independently.

exec() — Node Runtime

Node code runs in a persistent vm.Context (session). MCP imports like import('mcp/github') are intercepted by Node.js loader hooks, which return generated shim modules. The shim calls the real MCP client via globalThis.__mcpClients. The IIFE return value becomes ExecResult.result.

What Claude sends ~120 tokens
exec() call
const { listPullRequests } = await import('mcp/github'); const { listIssues } = await import('mcp/linear'); const [prs, issues] = await Promise.all([ listPullRequests({ owner: 'acme', repo: 'api', state: 'open' }), listIssues({ teamId: 'ENG', state: 'started' }), ]); return { openPRs: prs.length, blockedIssues: issues.filter(i => i.priority === 1).length, staleReviews: prs.filter(p => p.draft === false && p.requested_reviewers.length > 0) .map(p => ({ title: p.title, url: p.html_url })), };
What Claude receives ~60 tokens
ExecResult
{"result":{"openPRs":7,"blockedIssues":2,"staleReviews":[{"title":"feat: add rate limiting","url":"https://github.com/acme/api/pull/142"},{"title":"fix: auth token refresh","url":"https://github.com/acme/api/pull/138"}]},"tool_calls":[{"server":"github","tool":"listPullRequests","duration_ms":312},{"server":"linear","tool":"listIssues","duration_ms":187}]}
▼ 95%+ reduction — raw PR/issue data never entered context
exec() Node flow — vm.Context + loader hooks
Generated shim (mcp/github virtual module)
// Loader hook generates this at import time // specifier: 'mcp/github' → url: 'virtual:mcp/github' export async function listPullRequests(params) { return globalThis.__mcpClients['github'] .callTool('listPullRequests', params); } export async function createPullRequest(params) { return globalThis.__mcpClients['github'] .callTool('createPullRequest', params); } // ... one export per tool in the catalog
Session persistence
Implicit session: Each conversation shares one vm.Context identified as __implicit__. Variables and functions defined in one exec() call persist to the next.
Explicit session: Pass session_id: "my-parallel-task" to isolate parallel workflows. Each session has its own vm.Context and __mcpClients binding.
Stateless note: Bash and Python runtimes are always stateless — each call is a fresh subprocess. Only the Node runtime maintains session state.

exec() — Python Runtime

Python code runs in a fresh uv run --isolated subprocess per call — no session state. MCP server access works via a generated mcp/ Python package written to /tmp. Each function in that package POSTs to the HTTP bridge, which proxies to the real MCP SDK clients. Stdout is the result.

Python with pandas + MCP imports
exec() call

exec() Python flow — uv + HTTP bridge
Generated mcp/salesforce.py
import json, urllib.request as _req _BRIDGE = "http://127.0.0.1:49821/call" _EXEC_ID = "a1b2c3d4-..." _SERVER = "salesforce" def _call(tool, **kwargs): data = json.dumps({ "server": _SERVER, "tool": tool, "args": kwargs, "execId": _EXEC_ID }).encode() with _req.urlopen(...) as r: body = json.loads(r.read()) if "error" in body: raise RuntimeError(body["error"]) return body["result"] def search(**kwargs): return _call("search", **kwargs) def get_record(**kwargs): return _call("getRecord", **kwargs)
Key design decisions
PEP 723 inline deps: The # /// script block at the top of Python code declares dependencies inline. uv run --isolated reads this and installs into a fresh venv before running. No global packages required.
Fresh process per call: Unlike Node, there is no Python session. Each exec() call spawns a new uv process, writes a tmp file, runs it, and cleans up. This means you cannot share variables between Python exec calls.
Bridge lifecycle: The HTTP bridge listens on a random loopback port decided at server startup. The port is injected into each generated package as _BRIDGE. The bridge flushes its tool_calls log per execId after the subprocess exits.

exec() — Bash Runtime

The simplest runtime. Bash code runs in a stateless subprocess via spawn('bash', ['-c', code]). No session, no MCP imports — stdout is the result. Best for shell pipelines, file operations, and chaining with data from a previous Node exec.

exec() Bash flow — stateless subprocess
Env filtering: By default, only a safe allowlist (PATH, HOME, TMPDIR, LANG, etc.) is passed to the subprocess. Add variables via sandbox.env.allow in settings.json, or pass them per-call via runtime: {type: "bash", env: {MY_VAR: "val"}}.
Timeout handling: Send SIGTERM on timeout. If the process doesn't exit within 2s, escalate to SIGKILL. Exit code 124 signals a timeout (matches the GNU timeout convention).
Cross-runtime data threading
// Step 1: Node fetches + exports data const { query } = await import('mcp/database'); const rows = await query({}); // Write to /tmp for Bash to read const fs = await import('node:fs/promises'); await fs.writeFile('/tmp/rows.csv', toCsv(rows)); return '/tmp/rows.csv written'; // Step 2: Bash processes it // runtime: "bash" // code: "awk '...' /tmp/rows.csv | sort | head -10"

Use ExecResult.stdout to thread raw data between runtimes. The agent writes it explicitly — no automatic piping.

Session Model

Sessions are Node.js vm.Context objects managed by SessionManager. They persist JavaScript state between exec() calls within a conversation. Max 100 concurrent sessions; 10-minute idle timeout.

Session globals injected at creation
// vm.createContext() receives: { // Safe Node.js globals Buffer, URL, URLSearchParams, TextEncoder, TextDecoder, structuredClone, queueMicrotask, atob, btoa, AbortController, setTimeout, clearTimeout, setInterval, clearInterval, setImmediate, clearImmediate, // mcp-exec internals __mcpClients: { github: { callTool: (name, params) => ... }, slack: { callTool: (name, params) => ... }, // ... one per connected server }, __session_id: "__implicit__", } // Deliberately excluded: // process, require, __dirname, __filename
Shim bridging: Because loader hooks run in the main context (globalThis), not the vm.Context, the Node runtime temporarily copies __mcpClients onto globalThis for the duration of each exec call, then restores the previous value. This bridges the two execution contexts.
Session manager — state transitions
Implicit vs explicit sessions
Implicit (default)
// No session_id — uses "__implicit__" // All calls in this conversation share one vm.Context await exec({ code: 'globalThis.x = 42', runtime: 'node' }); await exec({ code: 'return globalThis.x', runtime: 'node' }); // → 42 ✓ state persists
Explicit (parallel isolation)
// Parallel agents don't share state await Promise.all([ exec({ code: 'return fetchGithubMetrics()', runtime: 'node', session_id: 'agent-a' }), exec({ code: 'return fetchSlackDMs()', runtime: 'node', session_id: 'agent-b' }), ]);

Sandbox Layering

The OS-level sandbox wraps the entire mcp-exec process. vm.Context adds a second layer for Node code. Codex's platform sandbox (macOS sandbox-exec / Linux bwrap) adds an outer layer on top of all of it.

Sandbox layers — swimlane view
Sandbox config (settings.json)
{ "sandbox": { "network": { "allowedDomains": [ "api.github.com", "slack.com", "gmail.googleapis.com" ] }, "filesystem": { "allowWrite": ["/tmp", "~/.mcp-exec/sessions"], "denyRead": ["~/.ssh", "~/.gnupg"] }, "env": { "allow": ["GITHUB_TOKEN", "SLACK_TOKEN"] } } }

User settings (~/.claude/settings.json) and project settings (.claude/settings.json) are merged — arrays union, no overwrite.

What each layer enforces
LayerEnforces
srt (outer)Network domains, filesystem read/write, env var allowlist — OS-level, cannot be bypassed by code inside
vm.ContextNo process, no require, no __dirname — protects the host Node.js process from untrusted code
Codex platformmacOS sandbox-exec / Linux bwrap wraps the Claude Code process itself — adds a third outer ring
filterEnv()Subprocess env is filtered to the env.allow list before spawning bash or uv — secrets not in the allowlist can't leak into subprocesses

Hook Compatibility

Claude Code hooks fire on tool call events. When exec() runs code that calls downstream MCP tools internally, those calls happen inside the sandbox — Claude Code's hook system never sees them. This is by design: it's why intermediate results don't enter context.

What hooks see when exec() is used
Hook reference table
Hook event Fires on exec()? Fires on tools()? Notes
PreToolUse YES YES Fires for the exec or tools call itself. Can deny or modify args.
PostToolUse YES YES Fires after exec or tools returns. Sees ExecResult, not raw data.
PreToolUse (downstream) NO N/A Downstream calls happen inside the sandbox process. Hook system is in the Claude Code process — different process, no IPC.
PostToolUse (downstream) NO N/A Same reason. Downstream results never cross the process boundary into Claude Code.
Notification YES YES Fires normally — exec() and tools() are MCP tools like any other from Claude's perspective.
Stop YES YES Not affected by exec() behavior.
memtree / other MCP hooks PARTIAL YES If memtree intercepts tool calls, it sees exec and tools calls but not the downstream calls inside exec(). The ExecResult is recorded, not the intermediate data — which is the point.
Composability: mcp-exec works correctly alongside memtree and other hook-based plugins. memtree records the exec() tool call and its compact result — not 40k rows of raw data. This stacks the savings: mcp-exec eliminates intermediate data, memtree compresses the result reference.

Install

mcp-exec is a Claude Code plugin. Installing it registers the MCP server and primes the skill loader. No manual MCP config editing required.

1

Install the plugin

Requires Claude Code with the marketplace plugin.

/plugin install mcp-exec@joeblackwaslike
2

Prime the skill loader

Adds the activation rule to your CLAUDE.md so the model knows when to use tools() and exec().

/mcp-exec-prime-skill
3

Configure the sandbox

Add a sandbox block to ~/.claude/settings.json with the domains your MCP servers need.

{ "sandbox": { "network": { "allowedDomains": ["api.github.com"] }, "env": { "allow": ["GITHUB_TOKEN"] } } }

Verify connectivity
